#+TITLE: fussy — Performance Benchmark
#+OPTIONS: toc:2

* What this is

A self-contained benchmark comparing =fussy= against the
built-in =flex= completion style under identical conditions: same
Emacs, same =company-mode= pipeline, same lisp-interaction-mode test
buffer, same obarray-sized candidate pool.  The raw output is in
[[file:fussy-perf-report.txt][=fussy-perf-report.txt=]]; this file summarizes the findings.

The =flex= implementation is as of:

* How to reproduce

#+begin_src bash
  bash benchmark/fussy-perf-bench.sh
#+end_src

The script spawns two isolated =emacs -Q --batch= children — one for
fussy, one for flex — bootstraps each manually, runs the bench, and
concatenates the reports.  See [[file:fussy-perf-bench.sh][=fussy-perf-bench.sh=]]
and [[file:fussy-perf-bench.el][=fussy-perf-bench.el=]] for details.

* Per-keystroke timing

30 iterations per query, =company--fetch-candidates= timed in isolation.

| Buffer state     | fussy p50 | flex p50 | fussy speedup | fussy max | flex max |
|------------------+-----------+----------+---------------+-----------+----------|
| =(\vert)=        |   0.88 ms |  1.68 ms |          1.9× |   5.59 ms |  6.63 ms |
| =(s\vert)=       |   1.93 ms |  4.18 ms |          2.2× |   6.84 ms | 13.52 ms |
| =(se\vert)=      |   1.91 ms |  5.07 ms |          2.7× |   6.90 ms | 13.95 ms |
| =(set\vert)=     |   1.64 ms |  5.10 ms |          3.1× |   6.53 ms | 12.43 ms |
| =(setq\vert)=    |   1.31 ms |  5.20 ms |          *4.0×* |   6.16 ms |  5.32 ms |

Fussy wins on every column: median, p95, and tail max.  The tail max
gap is especially relevant — those are the keystrokes a user
actually perceives as a stutter.  Flex hits 13–14 ms tails for the
"narrowing" middle keystrokes (=s=, =se=, =set=); fussy stays under
7 ms across the board.

** Reading

Once the cache warms after the cold first keystroke, both styles cost
essentially the same (~60 µs per char — that's =all-completions= on
the prior cached list, not the style itself).  The entire delta is
the *first* keystroke.

Fussy's cold keystroke: *2.0 ms*. Flex's: *9.3 ms*. **4.6× speedup
on the keystroke that dominates the cost of typing a fresh prefix.**

* GC pressure

| Pass                 | fussy GCs | fussy GC ms | flex GCs | flex GC ms | flex / fussy |
|----------------------+-----------+-------------+----------+------------+--------------|
| per-keystroke (×30)  |         9 |       43 ms |       24 |     175 ms |     ~4× more |
| typing-sequence      |         2 |       10 ms |       15 |      86 ms |    *~8.6× more* |

Flex generates substantially more allocation pressure because it
allocates fresh propertized candidate strings on every keystroke
(=propertize= is 25 % of flex's CPU samples — see profiler below).
Fussy mutates faces onto the reused obarray strings via
=fzf-native-highlight-all=, so per-keystroke allocation stays
roughly constant after the initial vector return.

The =gc-cons-threshold= bump inside =fussy-company--fetch-candidates=
(defcustom =fussy-company-gc-cons-threshold=, default 128 MB) caps
mid-keystroke GC events.  Fussy's tail max is 5.5–6.9 ms across
every query — the previous mid-popup GC spikes that produced
40–50 ms outliers are gone.

* Where each style spends time

CPU profile, query =s=, 500 iterations, top of stack.

** fussy (1002 samples total)

#+begin_src
fussy-all-completions-v1                            996  95%
  fussy-filter-by-scoring                           996  95%
    fussy-outer-score                               965  92%
      fussy-fzf-score                               965  92%
        fzf-native-score-all  (C)                   964  91%
          vconcat              (C-side return vec)  448  42%
    all-completions          (C, obarray scan)       31   2%
fussy--sort (fzf--sort-highlight-advice)              6   0%
Automatic GC                                         46   4%
#+end_src

Almost the entire cost is the C-side fzf scoring call and the
vector return.  The Elisp wrapper, sort, postprocess, and
highlight are all rounding error.

** flex (1628 samples total — 1.6× more CPU than fussy)

#+begin_src
company-capf--candidates-1                         1185  58%
  completion-flex-all-completions                  1185  58%
    completion--flex-all-completions-1              655  32%
      propertize    (alloc fresh per candidate)     516  25%   ← biggest single cost
      all-completions (C, obarray scan)             125   6%
      completion--flex-cost (Gotoh DP)               10   0%
    mapcar add-face-text-property                   464  23%   ← eager highlight pass
company sort + postprocess                          443  22%
Automatic GC                                        381  18%
#+end_src

The dominant costs are *string allocation* (=propertize=) and *eager
highlight* (=add-face-text-property=).  Flex re-propertizes every
candidate on every keystroke; that's where the GC pressure comes
from.

* Summary

In this configuration — Emacs 31, =company-mode=, elisp obarray —
fussy is:

- *1.9–4.0× faster* at the keystroke median
- *2.0× faster* on the tail max for narrowing keystrokes
- *4.6× faster* on the cold first-keystroke of a typing burst
- *~9× lower GC time* during a typical typing sequence

The win comes from three places:

1. *fzf-native scoring is C* and returns the whole batch in one call;
   the Elisp loop overhead per candidate is gone.
2. *fussy mutates faces in place* on reused strings rather than
   propertizing fresh ones, slashing allocation pressure.
3. *=fussy-company-gc-cons-threshold=* defers GC out of the
   per-keystroke hot path, eliminating the 40–50 ms tail outliers
   the older bench used to show.

The raw report ([[file:fussy-perf-report.txt][=fussy-perf-report.txt=]]) has every per-call,
per-stage, and profiler number behind these summaries.
